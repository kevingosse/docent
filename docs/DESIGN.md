# Design: AI-Guided Code Review — *Trail* & the *Docent*

> **Status:** the review surface is built and the live Docent + authoring capture are wired. See
> `docs/STATUS.md` for the current state and what's left.

---

## 1. The problem

As developers delegate more implementation to AI agents, the bottleneck of software work is
shifting from *writing* code to *reviewing* it. Many developers already use their IDE almost
exclusively for review. But the review process hasn't adapted to agentic development, and the
mismatch is becoming acute.

The failure mode: I ask an agent to implement something. It does, autonomously — and hands back a
large diff that I now have to understand alone. That's heavy cognitive load, multiplied across every
PR. Both rational responses are bad: slow to a crawl, or (more common) cut corners and rubber-stamp.

The cost of rubber-stamping is **not** primarily that bugs slip through — modern coding agents are
good enough that this is survivable. The real cost is that **the human loses touch with the
codebase**. It's benign at first and compounds PR over PR, until the human can no longer reason about
code that is nominally theirs. That is an existential risk for the developer and for the organization
that depends on them.

## 2. What's actually lost

Before agentic development, a big complex change was reviewed by *sitting together*: the author
walked you through it. That walkthrough is far more than the diff. The author tells the **story** of
the change — why it's designed this way, what problem each layer solves, what was tried and abandoned,
where the bodies are buried.

That information only ever lived in the author's head, and it evaporated the moment the branch was
pushed. With a human author you could at least ask afterward. With an agent author, the context is
*discarded* at the end of the session — the story is gone unless we capture it deliberately.

The insight that follows:

> **The artifact we need to capture is precisely the information that cannot be reconstructed from
> the code itself.** The diff already encodes *what* changed; the compiler proves it's valid. What's
> missing — and irreplaceable — is the *why*: alternatives weighed and rejected, constraints that
> forced an awkward shape, assumptions baked in, the load-bearing thing that looks wrong but isn't.

## 3. Vision

Recreate the feeling of a peer walking you through their change — but with the author's reasoning
captured at the source, and a fresh, critical guide delivering it.

Two moments:

1. **Authoring.** While (and after) an agent implements a task, it records the decisions it makes,
   then synthesizes them into a structured narrative — the **Trail** — anchored to the actual diff.
2. **Reviewing.** The human opens the change in their IDE. A guide — the **Docent**, a live agent
   ideally running a *different* model for critical distance — walks them through it: bite-sized,
   opinionated, and ready at any moment to answer questions, debate alternatives, or verify a claim.
   The human drives the pace. The experience should be *engaging* — closer to a sharp colleague
   arguing with you about the code than to a wall of text or a compliance checklist.

The goal is not to catch bugs (though it helps). The goal is to **keep the human in touch with their
codebase** as more of it is written by machines.

## 4. North star & principles

- **Mental-model retention is the success metric.** Not bugs caught, not time saved. Did the human
  come away genuinely understanding the change? Every design choice serves this.
- **Carry only the non-reconstructable.** The Trail contains the *why*, not a restatement of the
  *what*. Never narrate the diff back to the reader.
- **Present 100%, hide nothing.** The Docent triages — but it never discards. Every change is
  presented in some form. Triage is *not* structured metadata: the Docent simply *says* how much a
  change is worth as it presents it ("generated by tool X — skip"; "metadata — skim for anything
  odd"; "this you must understand"), never reducing *visibility to zero*. You can't stay in touch
  with a codebase that's been hidden from you. Completeness is a guarantee.
- **The Docent presents; the human asks.** Questions from the AI to the human are noise. Engagement
  is created by the Docent taking *positions* about the code — a stance invites the human to agree,
  push back, or interrogate. The provocation lives in the content, not in a quiz.
- **Opinions must be calibrated.** Over-criticism is just a new flavor of noise; a Docent that flags
  everything trains the human to ignore it. Each objection must be earned and grounded in verifiable
  fact, and "this will bite you" must be kept distinct from "nitpick."
- **Engagement through agency, not gamification.** The fun comes from driving, uncovering, and
  debating — not points or badges. The human controls the pace; the Docent never autoplays.
- **Ground claims in truth.** The Docent's narrative is a set of *claims*; the diff (and the world —
  upstream repos, docs) is ground truth. Being harness-connected, the Docent can *act* to verify
  ("I diffed this vendored header against dotnet/runtime — identical, skip it"). Verification is what
  makes the opinions trustworthy.
- **Feel like the IDE, not a separate product.** Full structural navigation is retained everywhere.
  This is both a UX requirement and a strategic one (see [§10](#10-technical-decisions)).

## 5. Naming

The names set the tone of the whole product.

- **Trail** — the artifact (the captured + synthesized story). The Docent walks the reviewer along it
  section by section, and it pairs naturally with *trailhead* (where each section begins).
- **Docent** — the reviewing agent / persona. A docent is a museum guide who narrates a curated tour,
  presents rather than quizzes, and answers the questions you bring — an exact fit.

Rejected: "Story" (collides with Agile user stories).

## 6. Architecture

Three parts. A clean way to hold them: **script / performer / stage.**

### The Trail — the script
Produced by the coding agent. The triaged, regrouped-by-intent, narrated, flagged plan, anchored to
the diff, with a stated *reason* attached to anything down-weighted. Frozen once authored.

### The Docent — the performer
A live agent — ideally a *different* model than the author, for critical distance — connected through
a harness. It performs the script, answers what the human asks, and can take actions on demand:
verify a claim against the diff, diff a vendored file against upstream, pull in related code, follow a
reference. The script is fixed; the performer improvises around it.

### The plugin — the stage
The native JetBrains (IDEA / Rider) tool window that renders the Trail, hosts the Docent, and
integrates with the IDE's real editor and navigation.

### Connection: ACP
The plugin talks to the Docent over **ACP (Agent Client Protocol)** rather than binding to a specific
vendor's AI. This buys model independence (we *want* the freedom to choose a different, critical
model) and broad adoption (any ACP-capable harness). JetBrains AI is a fine first integration; ACP
keeps us from being locked to it.

Concretely:

- **We are the ACP *client*; the Docent is an agent subprocess we spawn** (e.g. the Claude Agent SDK
  ACP adapter, or Gemini CLI — swappable, which is the point). The first integration spawns a real
  *harness* agent, so it brings its own tools.
- **Hand-rolled minimal client, not the official Kotlin SDK.** JSON-RPC-over-stdio (NDJSON):
  `initialize` → `session/new {cwd}` → `session/prompt` → stream `session/update` chunks; grow the
  surface as needed. Rationale: JetBrains' own workbench hand-rolls it the same way; the platform's
  reusable ACP runner (`EelAcpProcessHandler`) is internal / Ultimate / newer-build, so we can't lean
  on it; hand-rolling keeps us platform-clean (since-build 243, loads in Rider/IDEA-CE) with **zero new
  dependencies** (Gson is already present). The official SDK is a fallback if we ever want full v1
  coverage cheaply.
- **Read-only review posture.** The Docent must never mutate the code under review. We mirror the
  workbench's own AI-review policy: auto-approve read/search/execute tool calls, reject
  edit/delete/move.

### Two agents, two roles

The author of the change is the live agent our UI talks to **directly** — seeded by the Trail it
produced. It knows the *why* first-hand, so its answers are authentic rather than speculation, and it
owns the change, so the feedback loop is native. Critical distance moves to a *second, optional* agent.

- **The Docent — the coding agent (author).** The live agent the human converses with: it walks them
  through the sections, answers questions with first-hand knowledge, and takes their remarks. **During the
  review it is read-only** — it must *queue* requested changes, never implement them mid-review
  (enforced by explicit instruction *and* by our edit-rejecting permission posture). On **"Complete
  review"** the queue is dispatched and it implements the changes. (Connecting to the human's *actual*
  workbench coding-agent session is the productionized form; near-term we spawn the agent ourselves and
  talk to it directly.)
- **The Reviewer / Critic — optional, a _different_ model.** Spawned only when the human asks for a code
  review. **Read-only, always.** Drops calibrated inline comments and may reply to the author's Trail
  comments; this is where "critical distance" (§8) and the audit mechanic (§7) live. It *advises* — it
  never owns the queue. The human decides which of its concerns become requested changes.

So the workbench integration still reduces to the *authoring side*: the human's coding agent must
(a) **document its decisions into a Trail** as it works, and (b) **hand the change-set off for
review** when done. That rides MCP (below), touching no workbench code.

### Integration with the JetBrains agent workbench

JetBrains ships `agent-workbench` (a "bring your own agent" workbench) and, inside it, an AI code
review that is today a **flat, non-interactive list of findings** — so the *experience* lane is wide
open for us. We assume the human commands their coding agent through this workbench, and we layer on
top of it. **Forking the workbench is the last resort**; the realistic end-state is that this either
proves itself and gets folded into the workbench or it dies — so depending on the workbench's
(unstable, `@Internal`) APIs is an acceptable risk.

- **Primary seam — MCP.** The IDE runs an in-process MCP server with a public, dynamic extension point
  (`com.intellij.mcpServer.mcpToolset`: implement `McpToolset`, annotate methods `@McpTool`). The
  workbench *always* wires its launched agents to that same IDE server (it generates `.awb/awb-mcp.json`
  pointing at the server URL, passes `--strict-mcp-config`, and merges the user's own servers). So any
  tool we publish is reachable by **whatever** agent the workbench launches — provider-agnostic, no
  fork. We ship this as an **optional module** gated on `com.intellij.mcpServer`, so the core plugin
  stays platform-clean.
- **Authoring capture (MCP).** Docent tools — e.g. `docent_record_decision(...)` and
  `docent_finalize_trail(...)` — that the coding agent calls as it works. The instruction to use them
  ("narrate the non-reconstructable *why*; hand off for review when a change-set is done") is injected
  *transparently* via the workbench's `sessionLaunchSpecAugmenter` EP and/or a shipped Docent
  skill + project guidelines, so the human never has to prompt for it.
- **Caller identity.** Inside a tool, `mcpCallInfo` exposes `clientInfo`, a `sessionId`, a custom
  `localAgentId`, and request headers — so a tool call can be mapped back to the exact workbench
  session that made it (especially via a `localAgentId`/env we set at launch).
- **Native-feel hooks (optional, workbench EPs).** Where polish is worth the coupling:
  `promptPaletteExtension` (a "Review with Docent" entry, exactly how their AI review hooks in),
  `sessionProvider` (Docent reviews as first-class threads in the Agent Threads tree),
  `mcpConfigProviderContributor` (inject MCP config), and `AgentPromptLauncherBridge` (a first-class
  "send a prompt to an existing session" API — our message-injection primitive).

### The review feedback loop

In the *productionized* form the Docent is the human's own workbench coding session, reached over MCP
rather than an instance we spawn — so the review's conclusions must flow *back* to it. How that
happens is **a leading hypothesis to validate when building, not a locked decision:**

- **Leading approach — the review *is* the tool's return value.** The coding agent calls a blocking
  handoff tool (`request_review()`); the tool opens our review UI (the separate Docent performs the
  critique, the human discusses/approves/requests changes), **blocks**, then **returns** the verdict
  and requested changes — to exactly the agent/session that called it, by construction. This needs no
  out-of-band injection and no session mapping, rides MCP's native request/response, and *also* answers
  "when does review happen": the agent hands off when it judges a change-set complete. (MCP supports
  long-running tool calls with progress heartbeats; because the tool runs inside the IDE it drives our
  own UI directly — no MCP elicitation needed.)
- **The risk to test first.** The IDE server imposes no timeout, but the *agent harness* may cap
  tool-call duration and kill a multi-minute block. If so, fall back to a **hybrid** (return a
  `reviewId` immediately; the agent polls `get_review_result(id)`) or to **out-of-band injection** via
  `AgentPromptLauncherBridge` (existing-task launch) using the mapped session identity.
- **Same vs. different model for the *discussion*.** The critique is always a separate model. The human
  *discussion* can be with the Docent (cheap/fast, no context pollution) with the conclusion returned
  to the coder, or continued with the original agent (which "knows what it did") once unblocked — the
  feedback loop keeps both open. Feeding the Docent the Trail recovers much of the "knows why"
  advantage without sharing the coder's context. To refine empirically.

## 7. The Trail (artifact)

### Dual generation: trace during, synthesize after
- **During** the task, the agent records *decision points* — not a reasoning dump, but the
  non-reconstructable moments: a choice between alternatives, a constraint hit, an assumption made, a
  surprise discovered, a verification performed.
- **After**, a synthesis pass organizes those real, logged events into a top-down narrative anchored
  to the final diff.

Pure "after" rationalizes and loses the genuine dead-ends; pure "during" is noisy and doesn't yet
know what mattered. Capturing decisions live and synthesizing late gives authenticity *and*
coherence — and structurally resists confabulation, because the narrative is built from a log of
things that actually happened, not from memory.

### Structure
A tree/DAG, not file order: *thesis → architectural moves → per-area detail → individual hunks.*
Regrouped by **intent**, which routinely crosses files and languages. Each node carries only what it
needs: a headline, the Docent's narration, and one or more code anchors. Importance and risk are *not* encoded as fields — the Docent conveys them
in the narration (see §9), keeping the file deliberately minimal.

### The audit mechanic
The Trail's narrative is a set of claims. The Docent (and ideally the synthesis pass too)
cross-checks claims against the diff and flags discrepancies: claims the diff doesn't support, risky
changes the story omitted, assumptions not actually guaranteed. The harness extends this to the
outside world (upstream sources, regenerating a generated file). This turns the "different model"
from a nice-to-have into a load-bearing auditor.

### Format
Deliberately unspecified for now. The file is read by the plugin, not by humans, so the format is an
implementation detail that should *emerge from the features* rather than be designed up front. It
must at minimum anchor narrative nodes to specific code (file + range, ideally a commit/blob hash so
anchors survive later edits).

## 8. The Docent (the coding agent, as live guide)

- **Live, not a static render.** Backed by an ACP agent that can act.
- **The author itself.** It knows the change first-hand, so it answers *why* with knowledge, not
  speculation — and it owns the change, so it can act on requested changes when the review completes.
- **Presents, doesn't ask.** Voiced as a sharp senior colleague with opinions — restrained, not a
  quirky mascot (this is a tool opened many times a day).
- **Verifies.** Establishes trust by acting on ground truth, not by asserting.
- **Keeps the human oriented**, surfacing where the change carries risk or uncertainty and steering
  attention budget there.
- **Connected as an ACP client we own** (hand-rolled; see [§6](#6-architecture)). **Read-only _during_
  the review** — it may read, search, and run to verify, and it *queues* requested changes rather than
  implementing them until the human completes the review.
- **The optional Reviewer/Critic** is a *different* model for genuine critical distance (§6): read-only
  always, advisory inline comments, never owns the change.

## 9. The review experience (plugin UI)

This is the crux of the product and where the most trial-and-error is expected. The following is a
**starting design to iterate from, not a finished spec.**

### The trailhead (opening)
The human must immediately get a thesis (what this change is and why) and a plan. From the number of
sections and their headlines, they should form an *instinctive* sense of the effort ahead — not an
explicit "5 minutes," but the gut feel you get from a book's table of contents. Triage is already
applied and visible: "61k lines changed; 60k are vendored — here are the 600 that are yours."

### The section (atomic unit)
A bite-sized step: a short, opinionated narration card + the anchored code + any risk flag + an "ask"
affordance scoped to this section + a manual advance. No autoplay — the human moves when ready. This is
the agency lever, and most of the gap between "I'm reviewing" and "I'm being lectured at."

### Triage is something the Docent *says*, not a field
Every change is reachable, and triage lives in the narration: when a change is low-value, the Docent
just says so as it presents it ("generated by this tool — nothing to read"; "verified identical to
upstream — skip"). There is no weight badge or score in the UI; importance comes through in *how* the
Docent talks about each change. Nothing is hidden — low-value changes are still presented, just
briefly.

### Author's voice, Docent's layer
When the Docent is live, the Trail's narration is the *coding agent's* voice (the author's account
of the change); we **preserve and show it**, and render the Docent's critique as a **distinct layer on
top** — its inline remarks and its take, visibly separate from the author's claims. Keeping "what the
author said" beside "what the critic thinks" is exactly where the audit mechanic lives
([§7](#7-the-trail-artifact)): the divergence between the two is the signal.

### Composed views
Beyond classic before/after diffs, the Docent can compose a synthetic view from fragments pulled from
*different* places — e.g., the deleted C# rescue monitor beside the deleted C++ state map beside the
new C++ guard that replaces them both, shown as one "this replaces that" section. Hard to do well, worth
the effort; it's a primitive no diff tool offers — and it's exactly why the substrate must use real
editors (each fragment stays fully navigable).

### Free exploration & resume
At any moment the human can break the flow — control-click into a function, wander off on their own
investigation — and then resume the Trail where they left it. Wandering is engagement, not
deviation; the Trail holds a resumable position the human can leave and return to. Following a
cross-file definition opens the target in the main editor pane, so the human ends up "wandering"
outside our tool window — the tool window must keep its place and offer a clear way back.

### Activity-adaptive narration (stretch)
Dwell time, pauses, backtracking, and self-directed navigation are honest, no-friction comprehension
signals that fit "present, don't ask." Use them to *adapt the narration* (slow down where the human
lingers, revisit what they returned to) — **never** to maximize time-in-app. The moment we optimize
engagement for its own sake, we've built a slot machine, not a review tool.

## 10. Technical decisions

- **Native IntelliJ Platform plugin (IDEA / Rider), from the start.** Not a webview. The code shown
  inside our canvas must be *real editors* (`EditorTextField` / embedded editor views) so that
  go-to-definition, structural navigation, and find-usages all work — including on fragments inside
  composed views. A JCEF/web canvas renders dead text and would force us to reimplement everything the
  editor gives for free.
- **We own the orchestration; the IDE owns the code.** The plugin renders the trailhead, sections,
  pacing, and composed-view layout; code cells are backed by the real editor.
- **Accepted trade-off: iteration speed.** Native Swing UI is slower to trial-and-error than web
  hot-reload, and navigation is the part we most want to iterate. We pay this cost deliberately,
  because (a) it's the only substrate that can express composed, navigable code — the novel, risky
  core — and (b) "feels like the IDE" is the whole strategic point. We blunt the cost by keeping the
  Trail as external data, so presentation iterates without regenerating content.
- **ACP for the agent connection** — model independence + adoption; JetBrains AI as an easy first
  integration, not a lock-in.
- **Hand-rolled ACP client, platform-clean core, integration as optional modules.** The live Docent
  connects over a minimal hand-rolled ACP client (no external SDK; see [§6](#6-architecture)); the
  agent-workbench and MCP integrations live in *optional* plugin modules gated on
  `com.intellij.agent.workbench` / `com.intellij.mcpServer`, so the core keeps loading in plain
  IDEA-CE / Rider at since-build 243 while the integration lights up on IDEs that bundle them. This
  mirrors how JetBrains structures the workbench itself (community core + Ultimate-only execution).

## 11. Non-goals

- Not a bug-finder or static-analysis tool (it surfaces concerns, but catching defects is not the
  metric).
- Not a replacement for the diff/VCS UI; it sits on top of the real change.
- Not a gamified, badge-driven experience.
- Not bound to one AI vendor or model.
- Not standardizing the Trail file format up front.

## 12. Glossary

- **Trail** — the captured + synthesized story artifact for a change (the *script*).
- **Docent** — the live agent that performs the Trail (the *performer*); **the coding agent / author
  itself**, talking to the UI directly: it guides, answers from first-hand knowledge, and queues
  requested changes (implementing them on "Complete review").
- **Reviewer / Critic** — the *optional, different-model* agent the human can summon for a code review;
  read-only, drops advisory inline comments, never owns the change.
- **Section** — the atomic unit of review: a bite-sized narration + anchored code + flags + ask.
- **Trailhead** — the opening overview: thesis + plan that conveys scope at a glance.
- **Triage / attention weight** — the Docent's labeling of each change by how much attention it
  deserves, always with a reason, never hiding anything.
- **Composed view** — a synthetic view stitched from code fragments across different files/languages.
